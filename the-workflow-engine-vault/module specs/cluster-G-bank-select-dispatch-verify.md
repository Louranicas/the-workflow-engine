## Cluster overview

Cluster G is the **librarian lane** of the workflow engine. It enforces a strict, non-negotiable pipeline:

```
m23 (Cluster F proposer)
  │  human review + curator approval
  ▼
m30  workflow_bank          — curated storage; sunset + escape surface
  │  diversity-enforced selection
  ▼
m31  selector               — n-gram + RALPH fitness-weighted decay pick
  │  crystalliser-consultation (P0 #7)
  ▼
m33  workflow_verifier       — 4-agent dry-run gate; TTL-guarded
  │  PASS or DEGRADED gates through; FAIL blocks
  ▼
m32  dispatcher             — Conductor wire; trap surface; escape display
  │  NEVER executes directly (P0 #3)
  ▼
HABITAT-CONDUCTOR           — sole execution authority
```

**Invariants that cannot be broken:**

- P0 #3: m32 NEVER executes a step directly. Every step goes through HABITAT-CONDUCTOR.
- P0 #7: m31 → m33 consultation before m32 dispatch (crystalliser-consultation-before-dispatch).
- P0 #9: m33 `last_verified_at` TTL gates re-runs. Stale = re-verify before dispatch.
- P0 #11: every `AcceptedWorkflow` carries an `escape_surface_profile`; m32 displays it before each step.
- P0 #15 (dependency): m32 runtime is gated on Conductor Wave 1B/1C/2/3 `auto_start=true`. Until that flip, m32 produces a hard startup refusal, not a silent no-op.

### CC-5: Substrate Learning Loop (G → H → back to F)

Dispatch events from m32 fan to Cluster H:

- `m40` (`nexus_event_emitter`) receives every `DispatchEvent`; SYNTHEX v2 Hebbian LTP/LTD reinforces or depresses pathway weights.
- `m41` (`lcm_rpc_client`) receives deploy-shaped workflow dispatch; routes through LCM 9-RPC supervisor.
- `m42` (`hebbian_feedback`) `POST /reinforce` to POVM with `workflow_trace_*` namespace.

These outputs eventually read back into m31's selection weights via stcortex pathway queries. The loop: m32 emits → H substrates learn → m31 reads updated weights → selection shifts → m20-m22 inputs shift over time.

### CC-6: Verification-Gated Dispatch (G internal: m33 → m32)

This is the only permitted path into m32. m32 checks `VerificationResult::last_verified_at` before constructing its Conductor message. If the TTL has expired, m32 returns `DispatchError::VerificationStale` and the caller must re-run m33.

### EscapeSurfaceProfile enum (Cipher constraint P0 #11)

This enum encodes which categories of privileged operations the workflow steps may reach. The classification is set by m23 at proposal time and is immutable after bank acceptance. m32 uses it for the display-before-step gate.

```rust
/// The highest-risk escape surface this workflow may touch.
///
/// Ordinal: ReadOnly < HostWrite < Network < SandboxEscape < Destructive.
/// A workflow touching BOTH HostWrite and Network steps carries `Network`
/// (the higher ordinal) as its profile.
///
/// Rationale: S102 openclaw incident (see hookify.preserve-blanket-guard.local.md).
/// Blanket operations dissolve named-exclusion safety. The escape surface profile
/// makes the risk class explicit and visible before each step.
#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum EscapeSurfaceProfile {
    /// Reads only: no filesystem writes, no network calls, no subprocess launch.
    ReadOnly,
    /// Writes to the host filesystem within the workspace boundary.
    HostWrite,
    /// Makes outbound network calls (HTTP, WebSocket, database, etc.).
    Network,
    /// Calls that may escape the sandbox boundary (docker exec, ssh, etc.).
    SandboxEscape,
    /// Destructive operations: delete, prune, --force, batch-rm, etc.
    Destructive,
}

impl EscapeSurfaceProfile {
    /// Returns a human-readable warning banner displayed by m32 before each step.
    #[must_use]
    pub fn step_banner(&self) -> &'static str {
        match self {
            Self::ReadOnly    => "[READ-ONLY] No writes or network in this step.",
            Self::HostWrite   => "[HOST-WRITE] This step writes to the host filesystem.",
            Self::Network     => "[NETWORK] This step makes outbound network calls.",
            Self::SandboxEscape => "[SANDBOX-ESCAPE] This step may exit the sandbox boundary. Verify intent.",
            Self::Destructive => "[DESTRUCTIVE] This step runs a potentially irreversible operation. Enumerate targets before proceeding.",
        }
    }
}
```

### Sunset semantics from m11

m11 (`engine_sunset_lifecycle`) runs an engine-wide sweep that:

1. Marks workflows past `sunset_at` as ineligible (does not delete; keeps audit trail).
2. Applies `ralph_decay_weight *= 0.98` per consolidation cycle for eligible workflows.
3. Reads `dispatch_count`, `last_verified_at`, and `accepted_at` to compute `freq × fitness × recency` — the RALPH fitness-weighted decay formula that m31 consumes.

m30 exposes `BankDb::apply_decay_tick(&self, now_ms: i64) -> Result<usize>` which m11 calls. m30 never initiates decay itself.

## m31 — `selector`

**Job:** Diversity-enforced selection from the bank. Applies n-gram similarity over step sequences (Command-3's librarian shape) with a rejection gate on high repetition, combined with RALPH fitness-weighted decay scoring. Outputs a ranked candidate list for m33 to verify.

**LOC estimate:** ~300
**Boilerplate lift:** `m10_pattern.rs` from memory-injection (~70%) — two-counter pattern, decay, three-tier equilibrium; `m39_fitness_tensor.rs` from orac-sidecar (~70%) — rolling-mean smoothing, 12D tensor infrastructure (subset used here); `povm-v2_lifecycle.rs` (~40%) — decay formula primitives.

### Selection algorithm

The selection function computes a composite score for each candidate workflow and applies diversity enforcement before returning the final selection.

**Step 1 — Candidate pool.** Fetch `BankDb::eligible(now_ms, limit=200)` (pre-sorted by `ralph_decay_weight DESC`). Workflows past `sunset_at` are excluded at the SQL layer.

**Step 2 — Composite score.** For each candidate:

```
score(w) = α · fitness_weight(w)
         + β · recency_score(w)
         + γ · frequency_penalty(w)
         + δ · diversity_bonus(w, recently_selected)
```

Where:
- `fitness_weight(w)` = `w.ralph_decay_weight` (1.0 → 0.0 over lifetime).
- `recency_score(w)` = `exp(-(now_ms - w.last_verified_at.unwrap_or(w.accepted_at)) / RECENCY_HALF_LIFE_MS)`.
- `frequency_penalty(w)` = `1.0 / (1.0 + w.dispatch_count as f64 * FREQ_SCALE)` — suppresses over-dispatched workflows.
- `diversity_bonus(w, recently_selected)` = `1.0 - max_ngram_similarity(w.steps, recently_selected_sequences)`.
- Default weights: `α = 0.40`, `β = 0.25`, `γ = 0.20`, `δ = 0.15`.

**Step 3 — N-gram similarity (Command-3 librarian shape).** The diversity gate uses bigram similarity over step-id sequences. Step ids are opaque tokens (not human-readable names, per F11). The similarity function:

```rust
/// Jaccard bigram similarity over step-id sequences.
/// Returns [0.0, 1.0]; 1.0 = identical sequences.
fn ngram_similarity(a: &[StepId], b: &[StepId], n: usize) -> f64 {
    let a_grams: HashSet<&[StepId]> = a.windows(n).collect();
    let b_grams: HashSet<&[StepId]> = b.windows(n).collect();
    if a_grams.is_empty() && b_grams.is_empty() { return 1.0; }
    if a_grams.is_empty() || b_grams.is_empty() { return 0.0; }
    let intersection = a_grams.intersection(&b_grams).count();
    intersection as f64 / a_grams.union(&b_grams).count() as f64
}
```

**Step 4 — Diversity enforcement (ORAC m40_mutation_selector algebra).** Adapted from the round-robin cycling and mono-parameter rejection logic in ORAC's mutation selector:

- **10-generation cooldown:** A workflow that was dispatched in the last 10 dispatch events is suppressed from selection (its diversity_bonus is set to 0.0 and composite score floored at MIN_SELECTION_SCORE).
- **50% mono-parameter rejection gate:** If more than 50% of the top-10 candidates share the same `lineage` id (same cascade cluster), the tail candidates from that lineage are dropped and replaced by the next-highest-scoring candidates from different lineages. This prevents a single well-performing cascade cluster from monopolising all selections.
- **Round-robin cycling across lineages:** When multiple lineages score similarly (within `LINEAGE_SCORE_TOLERANCE = 0.05`), the selector cycles through lineages in order of first-acceptance rather than always picking the highest scorer.

**Step 5 — Output.** Returns `Vec<RankedWorkflow>` (top-N, default 5) for m33 to verify. The top candidate is the recommendation; m33 may verify in order until it finds a PASS.

TTL-gates re-runs (stale = must re-verify). Reports `PASS`, `FAIL`, or `DEGRADED`. Added to fill the v1.2 allocation gap (was missing; added in single-phase).

**LOC estimate:** ~200

### Verification result schema

### TTL gating

`VERIFY_TTL_MS` defaults to 7 days (604_800_000 ms). m32 checks:

```rust
fn is_verification_fresh(wf: &AcceptedWorkflow, now_ms: i64) -> bool {
    wf.last_verified_at
        .map(|t| now_ms - t < VERIFY_TTL_MS)
        .unwrap_or(false)
}
```

If stale, m32 returns `DispatchError::VerificationStale { workflow_id, last_verified_at }` before constructing any Conductor message. The caller must re-run m33.

### Dry-run mechanism

m33 does not execute steps. It performs static analysis on the workflow definition:

1. **Mechanical gate (Wave 1):** Validate that all `StepDef` references resolve to known step types in the workflow-core type registry. Validate that `escape_surface_profile` is consistent with the step definitions (e.g., a `Network` step must not be in a `ReadOnly` profile workflow). Validate that `sunset_at` is in the future.

2. **4-agent gate (Wave 2, parallel):** Each agent analyses the `AcceptedWorkflow` struct + its `steps_json`. The agents are invoked as structured analysis prompts (in single-phase, these are sub-agent dispatches via the Task tool or equivalent; the spec leaves agent-dispatch mechanism open to the build-time implementation choice, subject to Zen audit at G7). Each writes to a `VerificationResult` slot.

3. **Gate decision (Wave 3, boolean AND):** All 4 agents must return APPROVE or APPROVE-WITH-NITS for PASS. Any REJECT yields FAIL. All APPROVE-WITH-NITS and no REJECT yields DEGRADED.

4. **Receipt write:** On PASS or DEGRADED, m33 calls `BankDb::record_verification(id, now_ms)` — the audit-first guarantee (write the record before returning the result to m32). On FAIL, no verification timestamp is written; the workflow remains unverified.

### Definition drift detection

m32 computes `FNV_hex(steps_json)` at dispatch time and compares against `VerificationResult::definition_hash`. If they differ (curator edited the workflow after verification), m32 returns `DispatchError::DefinitionDrifted`. The workflow must be re-verified with the current definition.

### Conductor wire protocol

m32 sends a `ConductorDispatchRequest` to the Conductor's dispatch endpoint. The shape is derived from `conductor_api.rs`'s HTTP surface (reference-only for v1.2, since HARD REFUSAL blocks an HTTP server in wf-dispatch itself), adapted to the Conductor's inbound contract:

```rust
/// Outbound message from m32 to HABITAT-CONDUCTOR.
///
/// Sent as newline-delimited JSON (NDJSON) to the Conductor's Unix socket
/// or HTTP endpoint (transport determined by Conductor Wave maturity).
/// Shape adapted from conductor_api.rs DeployRequest + m24_povm_bridge gold standard.
#[derive(Debug, Serialize, Deserialize)]
pub struct ConductorDispatchRequest {
    /// Unique dispatch id (UUID v7). Idempotency key.
    pub dispatch_id: String,
    /// The workflow being dispatched.
    pub workflow_id: WorkflowId,
    /// Resolved step list (immutable at dispatch time; from AcceptedWorkflow::steps).
    pub steps: Vec<ResolvedStep>,
    /// Escape surface of the workflow (Cipher P0 #11).
    pub escape_surface: EscapeSurfaceProfile,
    /// Unix ms of dispatch.
    pub dispatched_at: i64,
    /// Operator identity. Always "wf-dispatch/human" — never "agent" or "auto".
    pub operator: String,
    /// Verification receipt: last_verified_at + definition_hash.
    pub verification_receipt: VerificationReceipt,
    /// Dry-run flag. When true, Conductor logs but does not execute.
    pub dry_run: bool,
}

/// A single step as resolved at dispatch time.
#[derive(Debug, Serialize, Deserialize)]
pub struct ResolvedStep {
    /// Opaque step id (from StepDef).
    pub id: StepId,
    /// Step kind (e.g., "cargo_check", "binary_deploy", "health_probe").
    /// Human-readable for Conductor's audit log, but opaque in bank schema.
    pub kind: String,
    /// Step-level escape surface (may be lower than workflow-level profile).
    pub step_surface: EscapeSurfaceProfile,
    /// Trap annotations for this step (surfaced in the display-before-step gate).
    pub traps: Vec<TrapAnnotation>,
    /// Conductor-specific execution parameters (service, args, timeout).
    pub conductor_params: serde_json::Value,
}

/// Trap annotation for a single step, adapted from SKILL-forge trap encoding.
#[derive(Debug, Serialize, Deserialize)]
pub struct TrapAnnotation {
    /// Trap id (e.g., "T-URL-PREFIX", "T-POVM-WRITE-ONLY", "T-CURL-SF").
    pub id: String,
    /// Category: security, silent-failure, habitat-specific.
    pub category: String,
    /// How this workflow step surfaces or avoids the trap.
    pub mitigation: String,
}
```

### Resolved step list shape

Before constructing the `ConductorDispatchRequest`, m32 resolves `AcceptedWorkflow::steps` into `Vec<ResolvedStep>` by:

1. Looking up each `StepDef` in the workflow-core type registry.
2. Computing the per-step `step_surface` (the lowest surface that the step's params permit).
3. Loading trap annotations from the workflow-core trap registry (matched by step kind).

### Pre-dispatch gate sequence

m32 performs these checks in order before constructing the Conductor message. Any failure returns an error immediately.

```
1. probe_conductor()                         — Conductor live?
2. is_verification_fresh(wf, now_ms)         — m33 TTL (CC-6)
3. definition_hash_matches(wf, stored_hash)  — drift detection
4. wf.sunset_at > now_ms                     — sunset guard
5. dispatch_cooldown_ok(wf.id, now_ms)       — anti-storm (300s default)
```

### Fan-out to Cluster H

After the Conductor accepts the request, m32 emits a `WorkflowDispatchEvent` to Cluster H. This is fire-and-forget (non-blocking); a failure to fan out is logged but does not roll back the dispatch.


This event is the CC-5 entry point: m40 picks it up for SYNTHEX v2 NexusEvent emission, m41 for LCM RPC routing (deploy-shaped workflows only), m42 for POVM Hebbian reinforce.

```rust
fn init_dispatcher(cfg: &DispatcherConfig) -> Result<Dispatcher, DispatchError> {
    if !cfg.dispatch_enabled {
        // Log at ERROR level, not warn. This is a hard operational blocker.
        tracing::error!(
            conductor_addr = %cfg.conductor_addr,
            "CONDUCTOR_DISPATCH_ENABLED is not set — dispatcher initialised in REFUSE mode"
        );
        return Ok(Dispatcher::refuse_mode(cfg.conductor_addr.clone()));
    }
    // Live-probe the Conductor before accepting any dispatch requests.
    probe_conductor(&cfg.conductor_addr)?;
    Ok(Dispatcher::live_mode(cfg.clone()))
}
```

`Dispatcher::refuse_mode` returns `DispatchError::ConductorDispatchDisabled` on every `dispatch()` call. It never returns Ok(). This is not a TODO — it is the production behaviour until Luke flips the env var after Conductor Wave 3 bring-up.

## What stays forbidden (v1.2 hard refusals, single-phase carry-forward)

- m32 NEVER executes a step directly. Any code path that calls a `Command::new` or `tokio::process::Command` on a workflow step is a P0 violation.
- POVM writes outside `workflow_trace_*` namespace (AP30 collision avoidance applies to m42, not m32 directly, but m32 must not invent its own POVM writes).
- HTTP server in `wf-dispatch` binary (HARD REFUSAL). m32 communicates outbound to Conductor; it does not accept inbound connections.
- `use synthex_v2::*` — patterns are lifted from synthex-v2 sources, not imported.
- Self-modification of m46-m51 (AP27 hard boundary — Watcher scope).
